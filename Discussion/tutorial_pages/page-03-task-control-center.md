# Page 03: 輕量任務中控中心

## 這一頁要傳達什麼

這一頁的重點不是介紹所有表格名稱，而是讓使用者明白：Non-Blocking Life 是一個輕量任務中控中心，幫你在「現在要做什麼」與「突然想到什麼」之間快速切換，而不必自己每次重新整理整個系統。

使用者看完這一頁，應該要感受到：

1. 我不用每次都從零決定接下來做什麼。
2. 我有靈感或突發想法時，也不需要中斷主線去整理分類。
3. 日常流程其實可以很簡單：查詢候選、選一個開始、想到什麼先丟 inbox、之後再集中整理。

## 依照真實使用習慣收斂後的核心訊息

你目前最常做的兩件事，正好就是這一頁最該突出的內容：

1. action=query
由系統告訴你現在適合做的事有哪些，挑一個就進入下一頁的專注節奏。

2. action=add&sheet=inbox
無論是網頁、突發奇想、臨時念頭，先不分類、不判斷，全部快速記錄到 inbox。

接著，一天只需要固定一次打開 inbox 掃描與整理：

- 值得做的移去 tasks
- 比較像參考資料的移去 resources
- 還不確定該怎麼分類的，先留在 inbox

這比傳統任務系統一開始就要求精細分類，更符合「不中斷主線」的產品核心。

## 與 Page 04 的分工

這一頁負責回答：

- 我怎麼決定現在做什麼？
- 我突然想到事情時怎麼不中斷地收進系統？

下一頁 Page 04 才負責回答：

- 一旦選好任務，如何進入專注與休息節奏？

也就是說：

- Page 03 是中控與分流
- Page 04 是開始後的節奏管理

## 畫面構圖建議

### 主視覺概念

以一個中心控制台為主，左右兩條主要輸入流：

1. 左側是 query 流
系統列出幾個候選項，箭頭匯入中心，再由中心高亮其中一個，表示「現在最適合做這個」。

2. 右側是 inbox capture 流
網頁、燈泡、靈感、便條、連結等小圖示快速流入 inbox 盒子，表示「先收進來，不要卡住」。

3. 下方是每日整理流
inbox 底下往外分成 tasks、resources、keep in inbox 三個出口，代表每天集中掃描一次再分流。

### 畫面草圖

```text
  Query 候選        [ NBL 中控 ]         網頁/想法
  □ 候選A   --->       ◎           <---   🌐 💡 📝
  □ 候選B             /|\
  □ 候選C            / | \
                    tasks resources inbox
```

## 動畫結構建議

### 段落 1：Query 入口

- 左側出現 3 個候選任務卡片
- 中間的 NBL 中控亮起
- 其中一張候選卡被高亮或被框選
- 讓使用者理解：系統會先幫我整理當下可做的選項

### 段落 2：Inbox 快速收集

- 右側連續冒出 3 到 4 個輸入來源，例如 🌐 💡 📝 🔖
- 這些元素快速滑入 inbox 盒子
- inbox 盒子容量增加或堆疊增加
- 讓使用者理解：先記下來，不用現在就判斷

### 段落 3：每日一次整理

- inbox 打開
- 三個小出口依序亮起：tasks、resources、keep
- 幾個項目被分配到不同出口
- 讓使用者理解：整理不是每次發生，而是集中發生

## SVG 圖層建議

1. 中央中控台框體
2. 左側 query 候選卡片 3 張
3. 被選中的候選卡高亮層
4. 右側輸入來源小圖示
5. inbox 盒子
6. 下方三個分流出口
7. 分流標籤：tasks / resources / keep
8. 少量箭頭或路徑線

## 動畫時序

1. 0ms 到 500ms
中控台、左右區塊與 inbox 容器淡入。

2. 500ms 到 1800ms
左側 query 候選卡片依序出現，其中一張被選中。

3. 1800ms 到 3200ms
右側輸入來源依序飛入 inbox。

4. 3200ms 到 5000ms
inbox 打開，下方三個出口亮起，內容被分流出去。

5. 5000ms 到 5600ms
整個畫面停住，讓使用者有時間理解這是一個中控與分流中心。

## 視覺語氣

- 不要做得像繁重的企業管理系統
- 應該像一個輕量、清楚、有秩序的控制台
- 重點不是「複雜功能很多」，而是「你不必每次都重新決定或重新分類」

## 文案方向

這頁的文字建議聚焦這句：

你不用每次都重新思考下一步，也不用每個念頭出現時立刻整理分類。

中控中心負責兩件事：

- 幫你提出當下適合做的選項
- 幫你把突發想法先安全收進 inbox

- Page 01: 問題是主線被打斷

## 結論

Page 03 應該是整個 onboarding 的結構頁。它不是情緒頁，也不是操作頁，而是把產品工作流濃縮成「Query 推薦 + Inbox 收集 + 每日分流」三段，讓新使用者第一次真正看懂這個系統平常怎麼被用。
